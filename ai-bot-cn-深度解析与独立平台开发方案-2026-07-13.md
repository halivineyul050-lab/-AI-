# ai-bot.cn 深度解析与独立 AI 工具导航平台开发方案

> 分析对象：[https://ai-bot.cn/](https://ai-bot.cn/)
> 分析日期：2026-07-13（Asia/Shanghai）
> 分析方式：实际访问首页、导航与搜索、3 类分类页、3 个工具详情页、资讯/项目/教程/百科/专栏、应用商店、关于与广告合作页；同时抽查 HTML、响应头、robots、两套 Sitemap、静态资源和 390/768/1280px 布局。
> 时效说明：工具数量、广告、价格、文章与流量数据会变化；本文只把 2026-07-13 当日可观察状态写成事实。

## 证据标记

- `[事实]`：页面、HTML、响应头或交互中直接观察到。
- `[推测]`：由公开技术信号推导，不能当作站方披露。
- `[建议]`：为独立新平台提出的产品或技术方案，不代表原站实现。
- `[待验证]`：需 Search Console、GA、服务端代码、数据库或真实业务合同才能确认。

## 执行摘要

AI工具集不是普通的“网址链接墙”，而是以 AI 工具数据库为核心、以搜索和分类为发现入口、以工具详情长文承接 SEO、以资讯/教程/百科维持内容更新、以外链跳转完成价值交付，并通过广告、付费收录和效果营销变现的综合垂直门户。

核心判断如下：

1. `[事实]` 产品主链路为“首页/搜索/分类发现 -> 工具详情 -> 第三方官网”，部分商业卡片会跳过详情直接外链。
2. `[事实]` 首页同时承载左侧类目索引、顶部业务导航、多引擎搜索、AI快讯/项目/百科、热门/最新工具、17 个工具大类、广告和页脚链接云，信息密度高，但移动端页面高度约 24,792px。
3. `[事实]` 工具详情并非只有名称和链接，而是包含价格标记、分类、官网 CTA、介绍、功能、用法、价格/FAQ、截图、相似工具、评论和广告，具备完整的搜索落地页形态。
4. `[事实]` 内容体系已扩展为资讯、项目、教程、百科、专栏、问答、应用商店等多个内容面；“最新AI项目”归档当日显示 708 页，“AI教程”38 页，说明规模已远超小型导航站。
5. `[事实]` 广告合作页公开全站详情页广告、首页广告、CPC/CPA/CPS 效果广告、文章收录和网址收录；当日公开价分别为 99 元/篇与 399 元/网址，均宣称永久展示。
6. `[事实]` 技术上确认是 WordPress + OneNav 主题 + nginx + Cache Enabler，服务端输出完整 HTML，并接入 Google AdSense、GA4、百度统计和 WordPress AJAX。
7. `[事实]` SEO 基础较强：独立 Title/Description、详情页长文、丰富内链、更新时间和大量可索引页面；但 canonical、H1、结构化数据、Sitemap 完整性和重复 URL 处理存在明显缺口。
8. `[建议]` 新平台不应照搬“一页放完所有工具”的结构。更合适的差异化方向是：任务导向搜索、可解释筛选、统一工具数据模型、明确广告标识、编辑审核工作流和一方数据漏斗。

一句话产品模式：**以结构化 AI 工具库为底座，用搜索、分类和内容获取用户，再通过详情页外链导流与 B2B 推广服务变现。**

## 2.2 页面关系与导流

```text
搜索引擎 / 社交传播
        ↓
文章、百科、工具详情、分类页
        ↓  站内概念词 / 分类 / 标签 / 推荐卡
工具详情或相关内容
        ↓  主CTA
第三方工具官网

直接访问首页
        ↓
搜索 / 左侧分类 / 热门 / 最新 / 内容Tab
        ↓
分类列表或工具详情
        ↓
第三方工具官网
```

- `[事实]` 首页和分类页保留全站搜索与主导航，工具卡通常进入站内详情。
- `[事实]` 工具详情没有可见面包屑，但有分类按钮、正文概念内链、6 个相似工具和页脚热门链接。
- `[事实]` 文章页有作者、分类、发布时间、正文概念链接，侧栏提供热门工具、最新收录和最新文章。
- `[事实]` 少量推广卡直接外跳，并使用 `nofollow sponsored` 或推广参数；这缩短转化链路，却损失详情页阅读数据。
- `[事实]` 分类存在重叠：同一工具可以同时属于大类、子类和“常用工具”；首页分类、专题页分类、App 分类又是并行体系。

## 3.2 模块逐项分析

| 顺序 | 模块 | 内容与操作 | 产品/SEO/商业作用 | 判断 |
|---|---|---|---|---|
| 1 | 左侧分类栏 | 17 大类锚点，部分有二级入口；点击滚动到首页对应区 | 让高频用户快速定位，强化全站语义内链 | 保留；移动端改抽屉和可搜索分类 |
| 2 | 顶部导航 | 工具、App、资讯、项目、教程、关于；带下拉 | 分隔产品线并把首页流量分配到内容归档 | 保留；减少重复入口 |
| 3 | 品牌区 | 100px Logo；H1 为视觉隐藏的站名 | 建立品牌和首页关键词 | 保留，但 H1 应可访问且不依赖图片 |
| 4 | 超级搜索 | 站内与多外部引擎，按常用/搜索/社区/图片/生活分组 | 满足工具发现与通用检索；提高回访价值 | 核心保留；默认先做站内任务搜索 |
| 5 | 内容快捷Tab | AI快讯/项目/百科；默认输出图文卡，后两项异步加载 | 提供“还不知道工具名”的内容入口，制造新鲜度 | 保留；需明确每类定位和更新日期 |
| 6 | 双横幅广告 | 当日为两个 AI 产品横幅，小屏堆叠、大屏并排 | 直接商业收入；链接有 sponsored 标记 | 保留但加可见“推广”标签和频控 |
| 7 | 热门工具 | 12 张卡，Logo、名称、摘要 | 社会证明和快速决策 | 保留；热门算法与周期需透明 |
| 8 | 最新收录 | 6 张卡，有“新”标记 | 展示更新能力，并交付付费收录曝光 | 保留；区分自然收录和商业收录 |
| 9 | 工具分类长列表 | 17 组，部分含子类Tab及“查看更多” | 首页最大 SEO 与发现资产，覆盖大量长尾词 | 桌面保留摘要，移动端默认折叠 |
| 10 | 页脚 | 品牌说明、收藏提示、栏目/热门工具/外站链接、邮箱、备案 | 内链、信任与友情链接 | 重构为有标题的分组，避免链接云混杂 |

首页工具卡字段为 Logo、名称、一句话摘要；常规卡进详情，少数商业卡直接外跳。`[事实]` 首页 HTML 约 738KB、包含 486 张图片与 17 个脚本；长 DOM 是性能和可维护性风险。

## 3.3 多搜索引擎结构

- 常用：站内、Bing、百度、Google、Perplexity。
- 搜索：还包含 YOU、360、搜狗、神马。
- 社区：Hugging Face、GitHub、飞桨、魔搭、和鲸、掘金、知乎。
- 图片：文生图/素材与创作社区入口。
- 生活：电商、菜谱、铁路、快递、旅行等通用入口。

`[建议]` 独立新站可以保留“外部引擎切换”，但不应让生活搜索稀释 AI 品牌。更有价值的是把输入理解为任务，例如“把会议录音变成纪要”，返回推荐工具、适用平台、价格、中文支持和对比理由。

## 4.2 大量工具处理方式与问题

- `[事实]` 写作/视频通过 100 条一页再分页，图像页将 8 个子类和约 189 张卡放在同页。
- `[事实]` 页面没有价格、平台、语言、是否登录、API、开源、商用许可等筛选。
- `[事实]` 没有排序控件，用户无法切换“最热门/最新/免费/编辑推荐”。
- `[事实]` 卡片摘要长度不一致，推广卡与普通卡的点击目标不一致。
- `[问题]` 类目越大，用户越容易退化为逐卡扫描；100 条/页也放大 DOM、图片和认知负担。
- `[问题]` 缺少面包屑和可见分类描述，用户难以理解大类边界。
- `[问题]` 标题口径与实际数量可能不同，例如写作 Title 标“80+”，页面已超过 100。

## 4.3 标准工具分类页模板

```text
面包屑
H1 分类名 + 动态数量 + 编辑说明 + 更新时间
任务子类 / 标签
筛选栏：价格、平台、语言、输入输出、API、开源、商用、国内可用
排序：综合推荐 / 热门 / 最新 / 评分
已选条件 + 清空
工具卡网格
  Logo / 名称 / 摘要
  价格徽标 / 平台 / 中文 / 核心能力
  编辑推荐或推广标识
  比较 / 收藏 / 查看详情
分页或“加载更多”
分类FAQ + 相关教程 + 相关分类
```

移动端建议一行 1 张“信息型卡片”或 2 张“紧凑卡片”可切换；筛选采用底部抽屉，排序用单独下拉，已选条件保持横向滚动。

## 9.2 商业化的价值与风险

厂商愿意付费的原因不是“得到一个链接”本身，而是获得高意图 AI 用户、中文内容页、搜索曝光、首页新鲜度位和站内推荐网络。对用户而言，商业化只有在“帮助发现合适产品”时才有价值。

当前最明显的风险是商业内容与自然内容边界不清：

- `[事实]` 首页横幅本身未看到可见“广告/赞助”文字。
- `[事实]` 某商业落地稿同时出现在最新项目、教程、专栏、问答和项目/框架等归档首位，卡片外观接近普通编辑内容。
- `[事实]` 少数工具卡直接外跳，点击路径与普通卡不一致。
- `[建议/P0]` 所有商业卡、置顶位、联盟链接和稿件必须显示“推广/赞助内容”；自然排序与广告排序分开计算。

# 十一、响应式和移动端分析

## 11.1 实测结果

| 视口 | 导航与侧栏 | 搜索 | 卡片 | 主要问题 |
|---|---|---|---|---|
| 桌面1280+ | 220px固定侧栏，顶部完整导航 | 约800px宽 | 首页常见6列；分类按模板4–6列 | DOM巨大；信息密度高 |
| 平板768 | 侧栏压缩为约60px，顶部导航仍展开 | 约678px | 首页2列 | 内容主区约708px；文章详情另一实测中保留220px侧栏，正文仅约513px，断点不一致 |
| 手机390 | 侧栏隐藏，导航折叠为Logo+汉堡 | 约360px | 首页2列，每卡约193px含间距 | 首页高度约24,792px；滚动负担很高 |

进一步事实：

- 手机端工具卡保持2列，广告改为单列堆叠。
- 搜索类型和引擎列表横向滚动；内容Tab在小屏隐藏部分文字只保留图标。
- 详情页桌面侧栏在小屏隐藏，相关导流与广告没有等价迁移。
- 手机文章正文约15px/30px，阅读行距尚可；面包屑会换行。
- viewport 设置 `maximum-scale=1.0,user-scalable=no`，限制页面缩放，存在可访问性问题。

## 11.2 响应式改造建议

1. `P0` 在 `<992px` 彻底折叠全局侧栏，避免768px阅读区过窄。
2. `P0` 删除禁止缩放配置，保证无障碍缩放。
3. `P1` 首页每个大类只显示4–6个代表工具，剩余进入分类页；移动端大类默认折叠。
4. `P1` 将桌面右栏的相关工具、官网 CTA 和转化位插入移动正文中，而不是直接隐藏。
5. `P1` 广告素材按设备提供比例，预留固定高度避免 CLS。
6. `P1` 长列表采用分页或“加载更多”，保留可分享的页码 URL；不要无限滚动到无法返回位置。
7. `P2` 工具卡提供紧凑/详细视图切换，手机默认一列详细卡可提高可读性。
8. `P2` 按触控标准保证按钮最小44×44px，文本不小于14px。

# 十三、数据上报、PV和UV方案

## 13.1 指标定义

| 指标 | 定义 |
|---|---|
| 全站PV | 指定周期内所有有效 `page_view` 次数；排除机器人、监控、内部员工和重复发送 |
| 全站UV | 指定周期内去重 `user_id`；未登录用 `anonymous_id`，同一用户登录后做身份合并 |
| 页面PV/UV | 按标准化 `page_url/page_id` 聚合的PV/UV，去掉UTM等非内容参数 |
| Session | 30分钟无活动则新会话；跨午夜不强制切断，来源取首个有效触点 |
| 工具详情访问UV | `tool_detail_view` 按 `tool_id + user/anonymous_id` 去重 |
| 官网跳转次数 | `tool_official_click` 事件次数 |
| 官网跳转UV | 指定周期按 `tool_id + user/anonymous_id` 去重的官网点击人数 |
| 搜索次数/UV | `search_submit` 次数 / 搜索用户去重数 |
| 分类点击次数 | `category_click` 次数，区分首页、侧栏、导航和详情标签来源 |
| 广告曝光/点击 | 满足可见性阈值的 `ad_impression` / `ad_click`；CTR=点击/曝光 |
| 工具收藏 | 成功创建收藏的 `tool_favorite`；取消另记动作 |
| 工具提交 | 成功提交 `tool_submit_success`，不能只统计按钮点击 |
| 文章阅读 | 文章 `page_view`、有效阅读UV、50%/90%深度、完成率 |
| 停留时间 | 前台可见且有心跳的活跃秒数；关闭/隐藏时结算，不能简单用下页时间相减 |

## 13.3 核心漏斗

```text
F1 首页访问UV
→ F2 搜索提交UV 或 分类点击UV
→ F3 工具卡点击UV
→ F4 工具详情访问UV
→ F5 官网跳转UV
```

计算口径：同一分析窗口内按 `user_id/anonymous_id` 去重，并要求事件时间有序。分支漏斗可以分别看搜索用户和分类用户。

- 发现使用率 = `F2 UV / F1 UV`
- 卡片点击转化率 = `F3 UV / F2 UV`
- 详情到达率 = `F4 UV / F3 UV`
- 官网跳转率 = `F5 UV / F4 UV`
- 全链路转化率 = `F5 UV / F1 UV`
- PV口径同理，但 PV 漏斗允许一人多次，主要用于内容/组件效率，不代表人数转化。

应同时按来源、设备、新老用户、类目、价格、推广/自然、页面模板分层。广告卡直接外跳时进入独立漏斗“卡片曝光 -> 外链点击”，不要伪装成详情漏斗的一部分。

# 十四、SEO策略分析

## 14.1 实际观察到的SEO设计

| 项目 | 事实观察 | 评价 |
|---|---|---|
| Title | 首页、分类、工具、文章均有独立Title；分页自动带页码 | 基础较好；部分数量文案已陈旧 |
| Meta Description | 主页面均有描述；工具/文章根据内容生成 | 较好；分类分页存在复用首页描述 |
| H1/H2/H3 | 首页1个视觉隐藏H1；部分分类无H1；工具详情H1+H2较合理；部分教程多H1 | 不一致，影响语义与可访问性 |
| URL | 工具为 `/sites/{id}.html`，文章多用英文slug，分类为 `/favorites/{slug}/` | 可抓取，但工具ID URL不利品牌记忆 |
| Canonical | 工具和文章有；首页、分类、分类分页无 | 重大缺口 |
| 面包屑 | 文章详情有；分类和工具详情未见 | 工具发现链路不足 |
| 内部链接 | 首页类目、正文概念、分类按钮、相似工具、相关文章、上下篇、侧栏和页脚均有 | 是该站SEO的核心优势 |
| 图片Alt | 首页/分类大部分有；工具详情与文章仍有较多空Alt | 需按媒体用途修复 |
| 结构化数据 | 抽样首页、分类、工具、文章均未见JSON-LD/microdata | 缺失明显 |
| 搜索页 | 动态Title，`noindex, follow`；10条/页 | noindex合理；空状态弱 |
| 分页 | 有独立URL和页码；未见`rel=next/prev`、分类canonical | 基本可抓取但重复治理不足 |
| 更新时间 | 工具/文章OG包含发布时间与修改时间，页面也显示相对时间 | 有利于新鲜度，但应提供绝对日期 |
| 404 | 正确返回HTTP 404，不是软404 | 良好；建议显式noindex |
| Sitemap | 自定义与WordPress原生两套同时公开 | 数量严重不一致，见下文 |
| Robots | 屏蔽后台、搜索、部分路径，也屏蔽多种AI抓取器 | 需与品牌的AI搜索曝光目标一致 |

## 14.2 URL、Canonical和Sitemap重点问题

- `[事实]` `https://ai-bot.cn/sitemap.xml` 当日为单文件约5,068个URL；robots只声明这一份。
- `[事实]` `https://ai-bot.cn/wp-sitemap.xml` 同时公开约9,380个URL，其中文章约7,077、工具约1,673、App约552。
- `[事实]` 自定义 Sitemap 只含4个 `/sites/` 工具URL，而原生 Sitemap 含约1,673个，存在大量公开页面未被 robots 声明的主动提交文件覆盖。
- `[事实]` 分类带尾斜杠和不带尾斜杠版本均可返回200；分类又没有canonical，构成重复URL风险。
- `[事实]` 工具和文章canonical能剥离UTM回到干净URL；首页与分类没有同等保护。

## 14.3 建议SEO模板

| 页面 | Title模板 | H1 | Schema | Canonical |
|---|---|---|---|---|
| 首页 | `{品牌} - 按任务发现AI工具` | 品牌+价值主张 | WebSite + SearchAction + Organization | 根域唯一URL |
| 分类 | `{分类名}：{数量}款工具、价格与适用场景` | 分类名 | BreadcrumbList + ItemList | 当前规范分类URL |
| 工具详情 | `{工具名}：功能、价格、用法与替代品` | 工具名 | SoftwareApplication + FAQPage + BreadcrumbList | 工具slug URL |
| 文章 | `{文章标题} - {栏目}` | 文章标题 | Article/NewsArticle + BreadcrumbList | 文章slug URL |
| App详情 | `{App名}下载、平台与功能` | App名 | SoftwareApplication/MobileApplication | App slug URL |
| 搜索 | `“{query}”搜索结果` | 搜索结果 | 不输出 | noindex, follow |

Slug规则建议：小写英文/拼音，单词用连字符，发布后不随Title频繁改变；重命名必须301。分类分页 canonical 指向自身页，而不是全部指向第一页；过滤参数只有具备独立搜索价值的组合才允许索引，其余 noindex + canonical 到基础分类。

## 14.4 结构化数据与内容质量

1. 首页：`WebSite`、`SearchAction`、`Organization`。
2. 分类：`BreadcrumbList`、`ItemList`，列表项指向规范工具URL。
3. 工具：`SoftwareApplication`，只输出真实可见的价格、平台、评分；没有用户评分就不要伪造 `aggregateRating`。
4. 内容：`Article/NewsArticle`，包含作者、发布日期、修改日期、主图与来源。
5. FAQ：只有页面可见且由编辑维护的FAQ才能输出 `FAQPage`。
6. 图片：Logo用工具名Alt；装饰图空Alt；截图Alt描述实际界面；避免所有图片机械复制标题。

内容更新必须有原因：产品价格、可用性、功能、官网状态变化时更新并记录 `last_verified_at`。批量改“更新时间”但不改正文会损害可信度。

## 14.5 外链与Core Web Vitals

- 外链分为自然引用、官网主链接、联盟链接、广告链接；分别使用普通链接、`nofollow`、`sponsored`，并保留安全属性。
- 项目官网/GitHub应是可点击CTA而不是正文纯文本URL，这同时提升可用性与出站埋点质量。
- `[事实]` 首页原始HTML约738KB、486张图片；分类约203KB/111图；每日快讯HTML约938KB。大量媒体虽有懒加载，超大HTML和DOM仍构成LCP/INP风险。
- `[待验证]` 真实LCP、INP、CLS必须用CrUX、Search Console或移动网络Lighthouse；源码体积只能说明风险，不能写成实测CWV结论。
- `[建议]` 首页每类只SSR首批代表工具，其余进入分类或按需加载；图片使用AVIF/WebP、明确尺寸、CDN变换；广告预留固定槽位。

## 14.6 仍需专业工具/权限确认

Search Console索引覆盖、真实查询词与点击率、外链质量、CrUX、Google处罚/人工措施、GA流量、百度收录、服务端日志爬虫频率、内容生产流程和转化收入均无法从公开页面确认。

| 图片存储 | 已确认/无法确认 | URL主要为同域 `/wp-content/uploads/YYYY/MM/`；是否对象存储回源无法确认 |
| API | 已确认 | `theme.ajaxurl` 指向 `/wp-admin/admin-ajax.php`，浏览、收藏、随机内容等使用WordPress AJAX action |
| REST API | 已确认/无法确认 | `/wp-json/`等抽查返回首页HTML而非JSON；无法确认是插件关闭、重写还是安全策略 |
| 广告 | 已确认 | Google AdSense + 自营横幅/落地页；工具与文章模板加载广告单元 |
| 统计 | 已确认 | GA4 `G-KV0VT5CMJ6`、百度统计、OneNav postviews AJAX |
| 插件系统 | 已确认 | WordPress插件/主题体系；确认Cache Enabler，其他插件不能仅凭标记断言 |

## 15.1 请求、安全与性能事实

- HTTP会301到HTTPS，www会301到裸域，HSTS为一年。
- HTML支持gzip；CSS缓存约12小时，上传图片约30天。
- `server_name_session` Cookie出现在HTML/CSS/图片响应，HttpOnly、24小时，但抽查未见Secure/SameSite。
- 抽查未见CSP、X-Frame-Options、X-Content-Type-Options、Referrer-Policy、Permissions-Policy。
- 详情模板存在重复加载AdSense脚本的迹象，应合并为一次。

这些响应头和资源事实可以确认；WAF规则、服务器数量、数据库容量、备份、CI/CD、缓存命中率、源站IP和真实峰值QPS无法确认。

## 16.2 审核与监测机制

# 十七、网站优缺点

## 17.1 分维度评价

| 维度 | 优点 | 主要短板 |
|---|---|---|
| 产品定位 | AI工具发现明确，工具+内容+App形成矩阵 | 大量内容类型边界重叠 |
| 信息架构 | 17类覆盖广，入口丰富 | 首页过长，分类体系并行且重复 |
| 用户体验 | 卡片简洁，详情信息完整 | 分类无筛选/排序/对比，推广跳转不一致 |
| 搜索效率 | 站内+外部引擎、跨类型结果 | 仍是关键词匹配，空结果弱 |
| 内容质量 | 工具、教程、百科、项目形成内链 | 来源/作者透明度不足，商业稿混排 |
| 视觉设计 | 轻量、密集、系统字体、网格稳定 | 层级偏平，主CTA不突出，平板断点有问题 |
| 移动端 | 导航折叠、两列卡片、广告堆叠 | 首页约2.48万px，侧栏内容直接消失 |
| SEO | Title/Description、内链、详情长文强 | canonical/H1/Schema/Sitemap不完整 |
| 性能 | 有缓存、gzip和图片懒加载 | 首页/快讯HTML和DOM过大 |
| 商业化 | 收入方式清晰，B端交付产品化 | 广告/自然内容标识不足，信任风险 |
| 数据分析 | 接入GA4、百度统计和站内浏览AJAX | 公开页面无法看到统一事件、出站UV和漏斗体系 |
| 维护成本 | WordPress编辑效率高、模板复用 | 分类重叠、链接监测、陈旧文案和多内容线成本高 |

## 17.2 十个值得借鉴的设计

1. 工具实体不只给链接，而是建设长文详情与相似工具。
2. 首页搜索、分类、热门、最新和内容入口覆盖不同认知阶段。
3. 工具、App、文章是可交叉检索的多内容体系。
4. 教程/百科通过概念内链把内容流量导向工具。
5. 分类卡片保持Logo、名称、摘要的低学习成本。
6. 工具详情把官网CTA放在首屏附近。
7. 资讯、项目、百科、教程同时兼顾时效和常青搜索。
8. 稳定分页URL便于抓取、分享和回到位置。
9. 关于、合作、免责声明与备案建立基础信任。
10. 付费收录、广告位和效果营销形成清晰B端产品包。

## 17.3 十个明显问题

1. 商业内容和自然内容缺少醒目标识，且商业稿跨无关栏目置顶。
2. 分类页没有筛选、排序、比较与收藏，100卡/页扫描成本高。
3. 首页移动端高度接近2.5万px，17类全量展示不适合手机。
4. AI快讯、每日资讯、项目、框架、教程、专栏语义重叠。
5. 首页/分类缺canonical，分类尾斜杠双版本均200。
6. 两套Sitemap数量不一致，robots声明的版本遗漏大量工具URL。
7. 分类无H1、文章多H1、全站缺Schema，语义实现不统一。
8. 平板断点可能保留侧栏却隐藏详情右栏，正文宽度被挤压。
9. 无结果页只给“没有内容”，没有纠错或替代路径。
10. 首页、每日快讯和图像聚合页HTML/DOM过大，维护和性能成本高。

# 十八、类似网站的开发方案

## 18.1 规模判断与技术原则

公开Sitemap显示原站规模约万级URL、千级工具，不需要一开始就上复杂微服务和Elasticsearch集群。`[建议]` 新站先用模块化单体 + PostgreSQL全文搜索，验证数据质量、内容生产和转化，再按量升级。真正的难点是分类、审核、SEO、推荐与商业标识，不是QPS。

## 18.2 MVP范围

### 用户端

- 首页：品牌、任务搜索、热门、最新、精选分类、少量内容。
- 工具大类/子类页：过滤、排序、分页、可分享URL。
- 工具详情：结构化字段、截图、价格/平台、官网CTA、相似工具、更新时间。
- 搜索：工具/文章混搜、类型切换、零结果建议。
- 内容：教程/资讯两种起步模板、文章详情和关联工具。
- 产品提交：自然提交表单、邮件确认、状态查询。
- 关于、广告、隐私、条款、404。

### 管理端

- 工具/分类/标签/平台/价格管理。
- 提交审核、重复检测、链接检查。
- 文章编辑、关联工具、SEO字段、定时发布。
- 首页推荐位与基础广告位。
- 管理员角色、审计日志。
- 基础看板：PV/UV、搜索、卡片点击、详情、官网跳转。

MVP明确不做：复杂社交、评论等级、个性化推荐、厂商自助广告竞价、移动App、Elasticsearch、多地区微服务。

## 18.3 完整版本

- App商店与多平台版本。
- 登录、收藏、清单、工具比较、关注更新。
- 资讯/快讯/教程/百科/项目/问答/专栏完整内容矩阵。
- 自然语言任务搜索、语义召回、可解释推荐。
- 厂商工作台：认领工具、更新资料、提交活动、查看合规报表。
- 广告活动、创意、排期、频控、订单、发票、CPA/CPS回传。
- 自动链接监测、内容过期提醒、重复工具合并。
- 多语言、地区可用性与国际化SEO。
- A/B实验、推荐算法、订阅和消息中心。

## 18.4 推荐技术选型

| 层 | 推荐 | 理由 |
|---|---|---|
| Web前端 | Next.js + TypeScript + React Server Components | SSR/ISR、Metadata、Sitemap、图片优化和页面级缓存适合SEO站 |
| UI | Tailwind CSS或CSS Modules + Radix/shadcn基础组件 | 保持可访问性与独立视觉，不依赖现成导航主题 |
| API后端 | NestJS模块化单体 | 工具、内容、审核、广告、事件边界清晰，后续可拆服务 |
| 数据库 | PostgreSQL | 关系、JSONB、全文搜索、统计查询和约束能力足够 |
| ORM | Prisma或Drizzle | 类型安全迁移；团队已有偏好优先 |
| 缓存/队列 | Redis + BullMQ | 热门榜、限流、链接监测、异步通知和Sitemap任务 |
| 搜索MVP | PostgreSQL FTS + `pg_trgm` | 千级/万级实体足够，低运维成本 |
| 搜索扩展 | OpenSearch/Elasticsearch | 仅在>5万实体、复杂多字段召回/高QPS时引入 |
| 语义搜索 | pgvector或独立向量库 | 第二阶段用于任务理解，必须与结构化过滤混合 |
| 对象存储 | S3兼容对象存储 + CDN | Logo、截图、封面、响应式变体与生命周期管理 |
| 管理端 | 独立Next.js Admin，复用API与设计Token | 权限、审核、广告和数据工作台需定制 |
| 统计 | 自建事件网关 + ClickHouse（增长后）/PostgreSQL分区（MVP）+ GA4 | 一方漏斗与通用流量分析并存 |
| 监控 | OpenTelemetry + Sentry + Prometheus/Grafana | 错误、性能、任务与API可观测 |

内容管理不建议直接把工具数据塞进富文本CMS。工具、分类、链接、价格和平台必须是业务表；文章可使用TipTap/Markdown编辑器。若团队很小，可在MVP使用Headless CMS管理文章，但工具审核、广告和数据后台仍需独立业务端。

## 18.5 SEO实现方案

- 公开页默认SSR/ISR；工具/文章发布或更新时按tag失效缓存。
- 使用框架Metadata API统一Title、Description、Canonical、OG。
- 生成分片Sitemap Index：tools、apps、articles、categories、static；每片限制数量并准确输出lastmod。
- 工具/文章/分类生成对应JSON-LD；Schema数据必须来自可见字段。
- 过滤页制定白名单索引策略；搜索、预览、后台、用户页noindex。
- 重定向表管理slug变更、合并工具和旧URL。
- 图片在上传时生成尺寸、WebP/AVIF和Alt待办；CI检查页面H1、canonical与Schema。

## 18.6 数据上报方案

浏览器先发送到同域 `/events`，服务端补齐时间、匿名ID校验、UA分类与IP哈希，再写队列。事件采用版本化Schema，客户端失败可批量重试；广告曝光采用IntersectionObserver；页面活跃时间用visibility + heartbeat；官网跳转先本地上报再直接导航，不使用明显延迟中转页。

数据保留建议：原始事件短期保留，聚合指标长期保留；IP只存加盐哈希；用户可删除账户与收藏；Cookie/本地标识按适用隐私规则取得同意并提供退出。不要把搜索原文无期限保存，尤其需过滤邮箱、手机号、密钥等误输入敏感信息。

## 18.7 部署、可靠性与安全

- 部署：Docker；Web、API、Worker分进程；托管PostgreSQL/Redis；对象存储+CDN；Nginx/云负载均衡。
- 环境：dev/staging/prod隔离，数据库迁移先备份，蓝绿或滚动发布。
- 安全：HTTPS/HSTS、CSP、CSRF、XSS清洗、SQL参数化、上传白名单/病毒扫描、SSRF防护、后台MFA、最小权限、密钥托管、速率限制。
- 内容安全：外链安全扫描、广告素材审核、评论反垃圾、版权/投诉流程、商业内容标识。
- 备份：数据库PITR、对象存储版本、每日备份和季度恢复演练。
- 可用性：SLO、健康检查、队列死信、链接监测限速、第三方失败不阻塞页面。

## 18.8 开发阶段与交付

以下以1名产品、1名设计、2名前端、2名后端、1名测试/运维兼职为参考，不是固定报价。

| 阶段 | 周期参考 | 交付 |
|---|---:|---|
| 0. 产品与数据定义 | 1–2周 | 品牌定位、用户旅程、分类树、字段字典、原型、事件字典、内容规范 |
| 1. 技术底座 | 2周 | 仓库、CI/CD、数据库、权限、对象存储、设计系统、监控 |
| 2. MVP核心 | 4–6周 | 首页、分类、详情、搜索、文章、提交、后台工具/内容管理 |
| 3. SEO与数据 | 2–3周 | Metadata、Schema、Sitemap、Redirect、事件网关、漏斗看板、性能基线 |
| 4. 审核与商业化 | 3–4周 | 提交审核、推荐位、广告位、订单、商业标识、链接监测 |
